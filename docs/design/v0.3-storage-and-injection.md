## 存储：纯文件系统

```
~/.doramemory/
├── second/          # 单条事件（单次工具调用、单条消息）
├── minute/          # 小任务片段（连续几轮相关对话）
├── hour/            # 一次完整 session
├── day/             # 日摘要
├── week/            # 周摘要
├── month/           # 月摘要
├── year/            # 年骨架
└── MEMORY.md        # 自动维护，agent 注入用
```

**设计原则**:
- 目录结构即记忆层级，无需解释
- 备份 = `cp -r`，迁移 = `mv`，版本管理 = `git init`
- 没有数据库，没有索引文件，没有隐藏状态

---

## Flashbulb Memory

不设独立目录，用 frontmatter 原地标记：

```markdown
---
importance: 0.95
flashbulb: true     ← cron 压缩时跳过此文件
fidelity: 1.0       ← 永远保持 1.0
---
```

**触发条件**: `importance >= threshold`（默认 0.9，用户可配置）  
**效果**: cron 扫描时跳过有 `flashbulb: true` 的文件，fidelity 永不衰减

---

## Fidelity 衰减

```
fidelity(n) = α^n

n = 压缩次数（写入时 n=0，每经过一个时间层级 +1）
α = 压缩系数（用户配置，与压缩模型能力强相关）
```

**完整链路下各层 fidelity**（α=0.88）：

```
second  →  minute  →  hour  →  day  →  week  →  month  →  year
 1.0    ×α  0.88   ×α  0.77  ×α 0.68  ×α 0.60  ×α 0.52  ×α 0.46
```

**实际使用时**：agent 从哪个层开始写，就从那层起算 n=0。
低频 agent 从 hour 起，高频 agent 从 minute 起，`write_level` 配置决定。

---

## 压缩状态追踪

文件本身即状态，无需额外数据库：

```
compressed: false  →  待压缩
compressed: true   →  已进入上一层，原始文件保留不删
```

cron 扫描逻辑：
```
找 {layer}/ 下所有 compressed=false 且日期 < 当前周期边界的文件
→ 聚合 → 调用压缩算子 → 写入上一层
→ 将已处理文件的 compressed 改为 true
```

## 注入：维护 MEMORY.md

不暴露 `GET /memory` HTTP 接口，DoraMemory 直接维护 MEMORY.md 文件。

**更新触发时机**:
- `/compress` 调用后（新记忆写入）
- cron 压缩完成后（摘要更新）
- UserPromptSubmit hook（Claude Code 等产品，每轮刷新）

**文件结构**：DoraMemory 只管 START/END 区块，不碰其他内容：

```markdown
# 用户自己的内容...

<!-- DORAMEMORY:START -->
=== 2026-04-07（今天）· 保真度 0.88 ===
上午：与团队确定 compress endpoint 作为写入触发点。
下午：完成接口设计，四个接口职责清晰。

=== 2026-04-06 · 保真度 0.88 ===
建立仓库结构，完成 v0.1 设计文档和调研报告。

=== 2026-W13（上周）· 保真度 0.77 ===
项目启动，确定层级 Pooling 核心思路。
<!-- DORAMEMORY:END -->

# 用户自己的其他内容...
```

**兼容各家 agent 产品**：

```
Claude Code   →  CLAUDE.md
Cursor        →  .cursorrules
Windsurf      →  .windsurfrules
自研 agent    →  配置文件路径
```

## 对外接口（最终）

```
POST /compress    { messages, session_id, target_tokens }
                  → 写入 {write_level}/ 目录，更新 MEMORY.md

POST /recall      { query, max_tokens? }
                  → 遍历各层目录，语义检索，返回 MemoryChunk[]

POST /remember    { memory_id, importance?, content? }
                  → hour 及以上只能改 importance
                  → day 及以上可以改 content
                  → importance >= threshold 自动设 flashbulb: true
```

`GET /memory` 降级为内部方法，不对外暴露。

---

## 完整数据流

```
写：  POST /compress
        → 写入 hour/（或 minute/，取决于 write_level）
        → 更新 MEMORY.md

压：  cron（内部，不暴露接口）
        hour → day → week → month → year
        → 每层压缩后更新 MEMORY.md

读：  agent 从 MEMORY.md 自动获得近期背景（时间驱动）
      agent 主动调 POST /recall 获得语义相关记忆

标：  agent 主动调 POST /remember 调整 importance 或修正内容
```

## Decision Log

### DDR-009: 层级改用时间单位命名
**决策**: second/minute/hour/day/week/month/year 替代 L1~L5。  
**理由**: 自解释，目录名即语义，无需文档说明。  
**日期**: 2026-04-07  
**状态**: Accepted

### DDR-010: 存储使用纯文件系统
**决策**: 目录 + Markdown 文件，无数据库，无索引文件。  
**理由**: 透明可检查，备份迁移简单，git 友好，无外部依赖。  
**日期**: 2026-04-07  
**状态**: Accepted

### DDR-011: flashbulb 改为 frontmatter 标记
**决策**: 用 `flashbulb: true` frontmatter 替代独立 flashbulb/ 目录。  
**理由**: 文件留在原来的时间层，位置信息不丢失，cron 跳过更自然。  
**日期**: 2026-04-07  
**状态**: Accepted

### DDR-012: 注入改为维护 MEMORY.md
**决策**: 不暴露 GET /memory 接口，直接维护 agent 产品约定的 memory 文件。  
**理由**: 各家 agent 产品已建立 memory 文件约定，顺着管道走，零额外集成成本。  
**日期**: 2026-04-07  
**状态**: Accepted
