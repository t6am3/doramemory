## 写入路径：双轨

```
主路径：FSWatcher
  监控 messages 文件变更 → 增量读取 → 写 second/

Fallback：POST /ingest
  框架主动推送（不落盘的场景）→ 写 second/

两路都经过去重：message_id 已处理 → 跳过
```

### POST /ingest

```typescript
Request: {
  session_id: string
  messages: [{
    message_id: string    // 全局唯一，用于去重
    role:       'user' | 'assistant' | 'tool'
    content:    string
    timestamp:  string
  }]
}
Response: {
  written:    number      // 实际写入条数
  skipped:    number      // 因去重跳过的条数
}
```

## 去重：message_id + Bloom Filter

同一条消息可能同时经由 FSWatcher 和 /ingest 进来，需要幂等处理。

```
~/.doramemory/
  index/
    message_ids.bloom     ← bloom filter，判断 message_id 是否已处理
```

**为什么用 bloom filter**：
- 几十万条 message_id 只占几 MB
- 查询 O(1)，写入 O(1)
- 允许极低概率的误判（false positive），代价只是偶尔漏写一条，可接受
- 比完整 hash set 节省 10x 空间

写入流程：
```
收到 message_id
  → bloom filter 查询：可能存在？
    → 是：跳过（可能是重复，宁可漏写）
    → 否：写 second/，更新 bloom filter
```

## Embedding 策略

### 哪些层级写 Embedding

```
second / minute / hour   →  只做 grep（关键词搜索）
day / week / month / year / core  →  写 embedding（语义搜索）
```

**原因**：
- 原始层：消息碎、量大、噪音多，grep 精准且快
- 摘要层：经 LLM 压缩蒸馏，语义密度高，值得 embedding；文件数量少，索引维护成本低

### 更新时机

**批量更新，不实时**：每次 cron 压缩生成新摘要文件后，对新文件计算 embedding 并写入索引。

```
cron 压缩完成
  → 新生成的 day/week/month/year 文件
  → 批量计算 embedding
  → 更新 ~/.doramemory/index/embeddings.db（SQLite + vector 扩展）
```

### Embedding 模型

本地小模型，不依赖云端：
```yaml
# config.yaml
embedding_model: nomic-embed-text   # 本地运行，via ollama
```

## Recall 排序机制

### 两阶段召回

```
第一阶段：召回候选
  grep（second/minute/hour）     → 关键词精确匹配
  embedding 相似度（day 及以上） → 语义模糊匹配
  两路结果合并为候选池

第二阶段：排序
  主排序：embedding cosine 相似度（grep 命中视为相似度 1.0）
  次排序：时间近度（同等相似度下，越近越靠前）
```

### 不做 LLM rerank

```
多一次 LLM 调用 → 延迟不可控
交给 agent 自己判断 → 更灵活

返回结果里 layer 字段本身就是语义信号：
  layer: hour   → 原始细节，高可信，可能需要 drill-down
  layer: month  → 高度压缩概要，agent 自己决定够不够用
```

### /recall 接口（更新）

```typescript
Request: {
  query?:       string         // 语义检索
  time_range?:  {
    from: string               // "2026-W10" / "2026-03" / "2026-04-07T14:00"
    to?:  string
  }
  max_tokens?:  number
}

Response: {
  chunks: [{
    content:      string
    layer:        'second'|'minute'|'hour'|'day'|'week'|'month'|'year'|'core'
    time_range:   { from: string, to: string }
    sources:      string[]     // 下层文件 ID，供 drill-down
    match_type:   'keyword' | 'semantic' | 'time'  // 命中方式
  }]
}
```

`match_type` 让 agent 知道这条结果是怎么找到的，辅助判断可靠程度。

## 最终系统全景

```
写入（双轨）
  FSWatcher：文件变更 → 增量读 → 去重 → second/
  /ingest：  框架推送 → 去重  → second/

压缩（后台 cron）
  second→minute→hour→day→week→month→year
  压缩时 LLM 识别 flashbulb → core/recent/
  recent/ 超限 → 元压缩 → core/identity.md
  新摘要文件 → 批量更新 embedding 索引
  每次完成 → 刷新 MEMORY.md

注入
  MEMORY.md 三层：identity + flashbulb + 近期时间流
  session resume → session_view 实时重建

检索（按需）
  /recall：grep（原始层）+ embedding（摘要层）→ 时间近度排序
  /remember：标记 flashbulb 或修正摘要内容
```


## Decision Log

### DDR-021: /ingest 作为 FSWatcher 的 fallback
**决策**: 保留一个推送接口，供不落盘的框架使用，两路写入经同一去重层。  
**理由**: 文件监控覆盖大多数场景，但不能假设所有框架都写文件；fallback 保证系统通用性。  
**日期**: 2026-04-08  
**状态**: Accepted

### DDR-022: message_id + bloom filter 去重
**决策**: 所有写入经过 bloom filter 去重，允许极低概率漏写，不允许重复写。  
**理由**: 重复写入会导致压缩时内容膨胀，影响摘要质量；漏写极少数消息代价可接受。  
**日期**: 2026-04-08  
**状态**: Accepted

### DDR-023: Embedding 只写 day 及以上层级
**决策**: 原始层（second/minute/hour）只用 grep，摘要层（day+）写 embedding。  
**理由**: 原始层量大噪音多，grep 足够；摘要层语义密度高，值得向量化；cron 后批量更新控制成本。  
**日期**: 2026-04-08  
**状态**: Accepted

### DDR-024: Recall 不做 LLM rerank
**决策**: grep + embedding 相似度 + 时间近度排序，不引入 LLM rerank。  
**理由**: LLM rerank 增加不可控延迟；layer 字段已向 agent 传递足够的信息密度信号，agent 自行判断。  
**日期**: 2026-04-08  
**状态**: Accepted
