## 补丁：sessions/ 元信息目录

v0.7 遗漏：session 的时间信息没有地方存，resume 判断无从依赖。

```
~/.doramemory/
  sessions/
    {session_id}.yaml    ← 每个 session 的轻量元信息
```

```yaml
# sessions/a3f9c2.yaml
session_id:     a3f9c2
source_file:    ~/.claude/projects/xxx/conversations/143022.jsonl
started_at:     2026-04-07T14:30:00
last_active_at: 2026-04-07T16:45:00
message_count:  47
compressed_to:  hour       # 已压缩到哪层，null 表示只在 second/
```

FSWatcher 每次处理新消息时顺手更新 `last_active_at`，纯写文件，零额外开销。

resume 判断：读 `sessions/{id}.yaml` → 比较 `last_active_at`，O(1)。

## 完整存储目录（终稿）

```
~/.doramemory/
├── second/              # 原始消息，一条一文件
├── minute/              # 小任务片段
├── hour/                # 完整 session
├── day/                 # 日摘要
├── week/                # 周摘要
├── month/               # 月摘要
├── year/                # 年骨架
├── core/
│   ├── identity.md      # 身份层，≤200 tokens
│   └── recent/          # 具体 flashbulb，最多 N 条
├── sessions/            # session 元信息，started_at / last_active_at
├── index/
│   ├── message_ids.bloom  # 去重
│   ├── embeddings.db      # day+ 层的向量索引
│   └── cursors/           # FSWatcher byte offset
└── MEMORY.md            # 对外注入文件
```

## 技术栈

**语言：TypeScript**

- MCP 生态以 TS 为主，`@modelcontextprotocol/sdk` 官方 TS SDK
- `npx` 分发天然支持
- 文件 IO、HTTP、调度全有成熟库
- 类型系统对多格式 message parser 很有帮助

### 核心依赖

| 模块 | 库 | 理由 |
|------|----|------|
| 文件监控 | `chokidar` | 跨平台，稳定，处理过原子写入、重命名等边界 |
| HTTP server | `fastify` | 轻量，TS 友好，性能好 |
| MCP server | `@modelcontextprotocol/sdk` | 官方 SDK |
| 定时调度 | `node-cron` | 轻量，不依赖系统 crontab |
| SQLite | `better-sqlite3` | 同步 API，嵌入式，无独立进程 |
| 向量检索 | `sqlite-vec` | SQLite 扩展，保持纯文件系统哲学 |
| Bloom filter | `bloom-filters` | 成熟实现，支持序列化到文件 |
| YAML | `js-yaml` | frontmatter 读写 |
| LLM - Anthropic | `@anthropic-ai/sdk` | 官方 |
| LLM - OpenAI | `openai` | 官方，custom endpoint 也走这个 |
| CLI 交互 | `@clack/prompts` | 现代 CLI 引导体验 |
| 进度条 | `cli-progress` | 冷启动 embedding 建索引时用 |

### 为什么 `sqlite-vec` 而不是独立向量库

- 向量存在 SQLite 里，和文件系统哲学一致（整个 `~/.doramemory/` 都是文件）
- 不需要运行 Chroma / Qdrant 等独立进程
- `embeddings.db` 可以直接备份、迁移

## v1 实现范围

设计功能较多，v1 只做核心链路，其余迭代。

### v1 必须有

```
✓ npx doramemory init（引导 + daemon 安装 + MCP 注册）
✓ FSWatcher 监控 Claude Code 格式（claude parser）
✓ 写入 second/（含 message_id 去重）
✓ cron：second → hour → day（三层，够用）
✓ MEMORY.md 生成（近期时间流，占位符替换）
✓ MCP server：memory_recall（仅关键词搜索）
✓ sessions/ 元信息维护
```

### v1 不做，后续迭代

```
✗ minute/ 层（low priority，hour 够用）
✗ week / month / year 压缩（数据积累够了再做）
✗ core/identity（需要足够 flashbulb 才有意义）
✗ embedding 向量检索（v1 用 grep，够用）
✗ /ingest HTTP 接口（FSWatcher 先跑通）
✗ session resume 视图重建（复杂，后做）
✗ Cursor / Windsurf parser（先只支持 Claude Code）
✗ OpenAI / custom LLM（先只支持 Anthropic）
```

### v1 的完整数据流

```
Claude Code 对话落盘
  → chokidar 检测
  → claude parser 解析消息
  → bloom filter 去重
  → 写 second/{timestamp}.md
  → 更新 sessions/{id}.yaml

cron 每日
  → second/ → hour/（聚合，LLM 压缩）
  → hour/   → day/（聚合，LLM 压缩，识别 flashbulb）
  → 刷新 MEMORY.md（identity + flashbulb + 近期时间流）

agent 读 CLAUDE.md
  → {{DORAMEMORY}} 占位符已被替换为近期记忆

agent 主动 recall（MCP）
  → grep 搜索 second/ + day/
  → 返回 MemoryChunk[]
```

## Decision Log

### DDR-028: sessions/ 目录存储 session 元信息
**决策**: 每个 session 独立 yaml 文件，存 started_at / last_active_at / compressed_to。  
**理由**: resume 判断需要 O(1) 读取时间信息，扫 second/ 文件代价太高。  
**日期**: 2026-04-08  
**状态**: Accepted

### DDR-029: 技术栈 TypeScript + sqlite-vec
**决策**: TS 实现，向量存 SQLite（sqlite-vec 扩展），不引入独立向量数据库。  
**理由**: MCP 生态以 TS 为主；sqlite-vec 保持纯文件系统哲学，无额外进程依赖。  
**日期**: 2026-04-08  
**状态**: Accepted

### DDR-030: v1 只做 Claude Code + Anthropic + 三层压缩
**决策**: v1 范围收窄到核心链路可跑通，其余功能迭代。  
**理由**: 先验证核心假设（时间层级压缩 + MEMORY.md 注入有效），再扩展覆盖度。  
**日期**: 2026-04-08  
**状态**: Accepted
