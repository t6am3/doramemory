# DoraMemory 设计方案 v0.4

**版本**: v0.4  
**日期**: 2026-04-07  
**状态**: 设计阶段终稿  
**变更摘要**: 去除伪精度指标、完善 flashbulb 生命周期、长存续 session 管理、消息 oversize 处理

---

## 核心设计思想

人类记忆对近期事件保真度高，对远期事件只保留骨架。将这一特性建模为**时间维度上的分层压缩**：

- 记忆按时间粒度分层存储，越老越压缩
- 压缩由 LLM 完成，带有认知偏好，不是机械摘要
- Agent 当前视角决定能看到什么——近期清晰，远期模糊
- 极重要的事件脱离时间轴，沉淀为身份认知

**设计原则**：DoraMemory 是数据库 + 检索引擎，不是中间件。不修改用户原始 prompt，不拦截 LLM 调用链。所有读写均为显式调用。

## 存储：纯文件系统

```
~/.doramemory/
├── second/          # 单条事件（工具调用、单条消息）
├── minute/          # 小任务片段（连续几轮相关对话）
├── hour/            # 一次完整 session
├── day/             # 日摘要
├── week/            # 周摘要
├── month/           # 月摘要
├── year/            # 年骨架
├── core/
│   ├── identity.md  # 身份层，≤200 tokens，极度抽象
│   └── recent/      # 具体 flashbulb 事件，最多 N 条
│       ├── 2026-04-doramemory-project.md
│       └── 2025-10-team-transition.md
└── MEMORY.md        # 自动维护，agent 注入用
```

**核心约束**：目录结构即记忆层级，无需解释。备份 = `cp -r`，迁移 = `mv`，版本管理 = `git init`。没有数据库，没有索引文件，没有隐藏状态。

## 文件格式

YAML frontmatter + Markdown 正文。只保留**有行为意义**的字段，去除伪精度数字。

**写入层**（second / minute / hour）：
```markdown
---
id: 2026-04-07-14
session_id: 143022
timestamp: 2026-04-07T14:30:22
flashbulb: false
compressed: false
---

[原始内容]
```

**摘要层**（day / week / month / year）：
```markdown
---
id: 2026-04-07
sources: [2026-04-07-09, 2026-04-07-14, 2026-04-07-16]
compressed_at: 2026-04-07T23:59:00
flashbulb: false
compressed: false
---

=== 上午 ===
与团队确定写入触发机制，/compress 简化为 /write。

=== 下午 ===
完成接口设计，确定三个对外接口。
```

**只保留的 frontmatter 字段**：

| 字段 | 用途 |
|------|------|
| `session_id` | session 感知查询、resume 时过滤 |
| `timestamp` | 时间排序、gap 检测 |
| `flashbulb` | 控制压缩行为（cron 跳过），布尔值 |
| `compressed` | 运维状态，cron 扫描用 |
| `sources` | drill-down 链路 |

**去除**：fidelity 系数、numeric importance、衰减公式。这些是伪精度，LLM 做语言推理不依赖数值，时间标记和内容详略本身已经传递了可靠程度。

## Flashbulb 生命周期

### 产生方式

Flashbulb 是压缩过程的副产物，由 LLM 在做 hour/→day/ 压缩时识别，不是人工打标签：

```
压缩输入：今天所有 hour/ 文件

压缩输出1：day/ 摘要（正常产物）
压缩输出2：flashbulb 候选（LLM 认为"这件事不该被遗忘"）
           → 回写 flashbulb: true 到对应源文件
           → 在 core/recent/ 创建索引文件
```

Agent 也可以通过 `/remember` 主动标记。

### 积累与管理

```
core/recent/  上限 N 条（默认 10）
              超出时，最老几条压缩进 core/identity.md

core/identity.md  硬限制 200 tokens
                  被新内容更新时由 LLM 重写
                  从"具体事件"抽象为"身份认知"
```

具体事件（flashbulb）随时间沉淀为抽象认知（identity），镜像人类记忆的工作方式：

```
"2025年10月 Alice 突然离开团队，项目陷入混乱"
           ↓ 时间久了
"经历过团队骨干离开的阵痛，知道如何稳住局面"
```

## 写入路径

### 正常写入：/write

```
POST /write {
  session_id: string
  messages:   Message[]
  action:     'append' | 'resume'
}
```

每条消息写成独立的 second/ 文件，**零 LLM 调用，纯 IO**。

**消息时间戳规则**：
- user message → 收到时间戳
- tool call / result → 调用 / 返回时间戳
- assistant message → 流式结束时间戳（思考+生成耗时本身也是信息）

**消息 oversize 处理**：

```
消息 size < threshold（默认 4k tokens）
  → 直接写 second/，无处理

消息 size >= threshold，内容重要（LLM 判断）
  → second/ 存原文（oversized: true）
  → 同时立即生成 minute/ 摘要（例外的 LLM 调用）
  → recall 优先返回 minute/ 摘要，drill-down 可取原文

消息 size >= threshold，内容是噪音（日志、文件列表等）
  → 截断写 second/（truncated: true）
  → 超出部分丢弃
```

### Resume 写入

`action: 'resume'` 时，检测 `last_active_at` gap。若 gap 超过阈值（默认 1 小时）：

```
Response 额外返回：
  session_view: string  ← 实时重建的 session-aware 视图
```

框架收到后自行注入 context，DoraMemory 不碰 context。

## 压缩链（内部 cron，不对外暴露）

```
second → minute → hour → day → week → month → year
                                ↓（压缩时识别）
                           core/recent/
                                ↓（积累过多时）
                          core/identity.md
```

**触发规则**：

| 层级 | 触发时机 |
|------|----------|
| second → minute | 当天结束 |
| minute → hour | 当天结束 |
| hour → day | 当天结束，同时识别 flashbulb |
| day → week | 每周一 |
| week → month | 每月1日 |
| month → year | 每年1月1日 |
| core/recent → identity | recent/ 超过 N 条时 |

**cron 扫描逻辑**：找 `{layer}/` 下所有 `compressed: false` 且日期 < 当前周期边界的文件 → 聚合 → 压缩 → 写上层 → 回写 `compressed: true`。

**Flashbulb 文件**：cron 跳过（`flashbulb: true`），原文永久保留，但其内容仍然会被包含在上层摘要中。

## Session 管理

### 普通新 session

```
session 开始 → 读 MEMORY.md（cron 维护，零延迟）注入
session 进行中 → 框架自己管 context，DoraMemory 不参与
session 结束 → /write 写入 second/，cron 异步压缩
```

**框架 context 管理与 DoraMemory 的分工**：

```
框架 context 满 → 框架自己做轻量 summarize（不走 DoraMemory）
                   DoraMemory 不管 context window 怎么滚动
                   只管 session 结束后的持久化写入
```

### 长存续 session（resume）

当用户在一个 session 暂停一周后回来：

```
不做 gap injection（人工扭曲，不自然）
而是：实时重建 session-aware 视图

视图结构：
  === 我们正在进行的对话（始于 2026-04-07）===
  [这个 session 的压缩历史，session 自身的连续性优先]
  
  === 近期背景（2026-04-08 至今）===
  [全局 MEMORY.md 的近期内容，其他 session 的记忆]
```

重建是一次 LLM 调用，有延迟，但只在 resume 时发生一次——符合预期，像人"想起来"的过程。

过滤逻辑：gap 期间其他 session 的记忆用 `session_id` 字段区分，不会把当前 session 自己的内容重复注入。

## 注入：MEMORY.md

DoraMemory 只管 START/END 区块。三层结构，每层有硬上限，总体积永远可控：

```markdown
<!-- DORAMEMORY:START -->

## 我是谁
在 AI agent 记忆领域工作，经历过团队变动，
擅长从混乱中找到系统性解法。
（来自 core/identity.md，≤200 tokens）

## 重要记忆
DoraMemory 项目启动，这是一次重要的方向转变。
...（来自 core/recent/，最多 N 条）

## 近期记忆
=== 昨天 ===
与团队确认 session resume 方案，不做 gap injection。

=== 上周 ===
完成 v0.3 设计，确定纯文件系统存储方案。

=== 上月 ===
项目启动，确定时间维度 Pooling 核心思路。

<!-- DORAMEMORY:END -->
```

**更新时机**：
- `/write` 调用后
- cron 压缩完成后
- UserPromptSubmit hook（Claude Code 等，每轮刷新）

**兼容各家 agent 产品**：

```
Claude Code   →  CLAUDE.md
Cursor        →  .cursorrules
Windsurf      →  .windsurfrules
自研 agent    →  配置文件路径
```

## 对外接口（最终三个）

### POST /write
写入原始消息，更新 MEMORY.md。

```typescript
Request: {
  session_id: string
  messages:   Message[]
  action:     'append' | 'resume'
}
Response: {
  written:       number
  session_view?: string   // 仅 resume 且有 gap 时返回
}
```

### POST /recall
语义检索 + 时间范围检索。

```typescript
Request: {
  query?:       string         // 语义检索（grep + embedding）
  time_range?:  {              // 时间范围检索
    from: string               // "2026-W10" / "2026-03" / "2026-04-07T14:00"
    to?:  string
  }
  max_tokens?:  number
}
Response: {
  chunks: [{
    content:      string
    layer:        string       // second/minute/hour/day/week/month/year/core
    time_range:   { from, to }
    sources:      string[]     // 下层文件 ID，供 drill-down
  }]
}
```

时间范围精度自动决定返回层级：问"上周"返回 week/ 文件，问"某天下午"返回 hour/ 文件。

### POST /remember
标记 flashbulb 或修正摘要内容。

```typescript
Request: {
  memory_id:  string
  flashbulb?: boolean     // 标记为永久保留
  content?:   string      // 仅 day/ 及以上可修改内容（摘要可被人工校正）
                          // second/minute/hour 只能改 flashbulb，内容是 ground truth
}
```

## Recall 搜索实现

```
A（grep）+ B（本地 embedding）组合：
  grep：快，适合关键词、人名、项目名等精确词
  embedding：慢，适合语义模糊的查询

策略：先 grep 过滤候选文件，再 embedding 排序
      embedding 用本地小模型（nomic-embed-text 等），不依赖云端
```

## 完整数据流

```
实时写：  对话消息 → POST /write → second/（纯 IO）

后台压：  cron 定时
          second→minute→hour→day→week→month→year
          压缩时识别 flashbulb → core/recent/
          recent 积累过多 → 元压缩 → core/identity.md
          每次压缩后刷新 MEMORY.md

注入：    session 开始，agent 读 MEMORY.md（已在 CLAUDE.md 等文件里）
          长存续 session resume，/write 返回 session_view 实时注入

主动召回：agent 调 POST /recall（语义或时间范围）
主动标记：agent 调 POST /remember
```

## Decision Log（v0.4 新增）

### DDR-013: 去除伪精度数值指标
**决策**: 移除 fidelity 系数、numeric importance、衰减公式。  
**理由**: LLM 做语言推理，不做数值计算。时间标记和内容详略已天然传递可靠程度，数字只是制造了精确感。  
**日期**: 2026-04-07  
**状态**: Accepted

### DDR-014: Flashbulb 由压缩过程产生
**决策**: Flashbulb 是 LLM 压缩时的副产物，而非阈值触发或人工打标签。  
**理由**: 重要记忆往往是事后才认识到重要的，压缩时有全局视角，判断更准确。  
**日期**: 2026-04-07  
**状态**: Accepted

### DDR-015: core/ 双层结构管理 flashbulb 积累
**决策**: core/recent/（具体事件，上限 N 条）+ core/identity.md（抽象认知，≤200 tokens）。  
**理由**: 具体 flashbulb 随时间沉淀为身份认知，镜像人类记忆工作方式，同时保证 MEMORY.md 体积有界。  
**日期**: 2026-04-07  
**状态**: Accepted

### DDR-016: 长存续 session 用 session-aware 视图替代 gap injection
**决策**: resume 时实时重建以当前 session 历史为核心的视图，不做 gap injection。  
**理由**: gap injection 是机器思维，人回来继续对话时感知的是对话自身的连续性，不是"补缺口"。  
**日期**: 2026-04-07  
**状态**: Accepted

### DDR-017: 框架 context 管理与 DoraMemory 分离
**决策**: 框架 context 满时自行处理，DoraMemory 只管 session 结束后的持久化。  
**理由**: DoraMemory 负责长期记忆，context window 管理是框架职责，两者混在一起导致阻塞和耦合。  
**日期**: 2026-04-07  
**状态**: Accepted
