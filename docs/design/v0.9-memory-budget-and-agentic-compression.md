# DoraMemory 设计方案 v0.9

**版本**: v0.9  
**日期**: 2026-04-08  
**状态**: 设计中  
**变更摘要**: 重新定义 MEMORY.md 生成模型——职责分离、分层固定预算、agentic 压缩校验

---

## 核心变更：DoraMemory 不管当前会话

v0.8 的 builder 从 second → year 所有层读取记忆，贪心填充 4k token。  
这有两个根本问题：

1. **职责不清**: 当前会话的消息本来就在 agent 的 context window 里，DoraMemory 重复注入是浪费
2. **近期吞远期**: 贪心算法下，day 级信息足以填满 4k，week/month/year 被彻底挤掉

### 新模型

```
Agent context window          DoraMemory MEMORY.md
┌─────────────────────┐       ┌─────────────────────┐
│ 当前会话消息         │       │ identity（我是谁）    │
│ (秒/分钟级)          │       │ flashbulb（重要事件） │
│ agent 自己维护       │       │ hour 层 × N 条       │
│ agent 自己压缩       │       │ day  层 × N 条       │
│                     │       │ week 层 × N 条       │
│                     │       │ month层 × N 条       │
│                     │       │ year 层 × N 条       │
└─────────────────────┘       └─────────────────────┘
        ↑ 实时                        ↑ cron 触发更新
```

**一句话**: 秒级/消息级记忆由 agent 自己维护，DoraMemory 只提供**当前会话之前**的历史压缩记忆。

## 分层固定预算

每层有独立的 **条目上限** 和 **每条 token 上限**，两者均可配置。层预算互不侵犯——day 再多也不会挤掉 month。

### 默认配置

```yaml
memory_budget:
  identity:
    max_tokens: 200          # 身份层，单条
  flashbulb:
    max_entries: 5           # 最多保留 N 条 flashbulb
    max_tokens_per_entry: 60 # 每条上限
  hour:
    max_entries: 6           # 最近 6 个完整小时
    max_tokens_per_entry: 200
  day:
    max_entries: 5           # 最近 5 天
    max_tokens_per_entry: 150
  week:
    max_entries: 4           # 最近 4 周
    max_tokens_per_entry: 100
  month:
    max_entries: 6           # 最近 6 个月
    max_tokens_per_entry: 80
  year:
    max_entries: 3           # 最近 3 年
    max_tokens_per_entry: 60
```

### 预算计算

| 层 | 条目 × 每条上限 | 层最大 tokens |
|----|----------------|--------------|
| identity | 1 × 200 | 200 |
| flashbulb | 5 × 60 | 300 |
| hour | 6 × 200 | 1200 |
| day | 5 × 150 | 750 |
| week | 4 × 100 | 400 |
| month | 6 × 80 | 480 |
| year | 3 × 60 | 180 |
| **总计** | | **3510** |

总上限确定性可控，不超过 ~3500 tokens。用户可通过调整配置控制总量。

### 层选择逻辑

同一时间段只从**最高压缩层**读取，不重复：

- 当天已完成的小时 → 从 `hour/` 读
- 昨天及更早 → 优先 `day/`，无则 fallback 到 `hour/`
- 上周及更早 → 优先 `week/`，无则 fallback 到 `day/`
- 上月及更早 → 优先 `month/`，无则 fallback 到 `week/`
- 去年及更早 → 优先 `year/`，无则 fallback 到 `month/`

## Agentic 压缩 + compress_as 工具校验

### 问题

之前的压缩流程：LLM 压缩 → 直接写入文件 → builder 读取时再 truncate。  
结果：truncate 可能在句子中间截断，破坏语义。

### 新流程

压缩改为 **agentic 模式**：LLM 不是直接输出文本，而是必须调用 `compress_as` tool 来提交压缩结果。tool 在写入前校验 token 限制。

```
压缩 hour/2026-04-08-13:
  LLM 读取 second/ 的原始消息
  → LLM 调用 compress_as(layer="hour", id="2026-04-08-13", content="...")
  → compress_as 检查:
     estimateTokens(content) <= config.memory_budget.hour.max_tokens_per_entry (200)?
     ✓ → 写入 hour/2026-04-08-13.md，返回 success
     ✗ → 返回错误: "超出限制: 当前 287 tokens，上限 200 tokens，请进一步压缩"
     → LLM 重新压缩，再次调用 compress_as
     → 最多重试 3 次，仍然超限则取最后一次结果强制 truncate 并警告
```

### compress_as tool 定义

```typescript
interface CompressAsInput {
  layer: 'hour' | 'day' | 'week' | 'month' | 'year'
  id: string           // 时间标识，如 "2026-04-08-13"
  content: string      // 压缩后的文本
  flashbulb?: boolean  // 标记为重要记忆
}

interface CompressAsOutput {
  success: boolean
  tokens_used: number
  tokens_limit: number
  error?: string       // 超限时的错误信息
}
```

### 优势

1. **写入时保证质量** — 不需要读取时截断，builder 直接拼接
2. **LLM 主动控制精度** — 错误信息告诉 LLM 还差多少，可以精确调整
3. **token 总量完全可控** — 每条文件都已经在限制内，拼接结果必然在预算内

## 对 builder.ts 的影响

builder 逻辑大幅简化，不再需要：
- ~~tokenCapForAge()~~ — 已在压缩时控制
- ~~truncateToTokens()~~ — 文件已经在限制内
- ~~贪心预算分配~~ — 每层独立读取

```
新 builder 伪代码:

1. 读 identity → 拼接
2. 读 flashbulb → 取最近 N 条 → 拼接
3. hour: 取最近 max_entries 个文件 → 拼接
4. day:  取最近 max_entries 个文件（排除已有 hour 的时段）→ 拼接
5. week: 取最近 max_entries 个文件（排除已有 day 的时段）→ 拼接
6. month/year: 同理
7. 输出 MEMORY.md
```

## 对 config 的影响

`DoraConfig` 新增 `memory_budget` 字段：

```typescript
interface LayerBudget {
  max_entries: number
  max_tokens_per_entry: number
}

interface MemoryBudgetConfig {
  identity: { max_tokens: number }
  flashbulb: LayerBudget
  hour: LayerBudget
  day: LayerBudget
  week: LayerBudget
  month: LayerBudget
  year: LayerBudget
}
```

全部字段有默认值，用户只需覆盖想改的部分。

## Decision Log

### DDR-031: DoraMemory 不注入当前会话消息
**决策**: MEMORY.md 只包含 hour 及以上的历史记忆，当前会话的秒/分钟级消息由 agent 自行维护。  
**理由**: 当前会话消息已在 agent 的 context window 中，重复注入浪费 token 且职责不清。  
**日期**: 2026-04-08  
**状态**: Accepted

### DDR-032: 分层固定预算，互不侵犯
**决策**: 每层有独立的条目上限和每条 token 上限，层之间预算不共享、不溢出。  
**理由**: 贪心共享预算会导致近期信息吞掉远期信息，违背"人类总能记住远期大概发生了什么"的认知模型。  
**日期**: 2026-04-08  
**状态**: Accepted

### DDR-033: Agentic 压缩 + compress_as 工具校验
**决策**: LLM 压缩时必须通过 compress_as tool 提交结果，tool 校验 token 限制后才写入。  
**理由**: 将质量控制前移到写入时，确保每条文件都在限制内，builder 读取时无需截断，总 token 完全可控。  
**日期**: 2026-04-08  
**状态**: Accepted

### DDR-034: 预算参数全部可配置
**决策**: max_entries 和 max_tokens_per_entry 均暴露为配置项，提供合理默认值。  
**理由**: 不同用户的使用强度和 context window 大小不同，硬编码不够灵活。  
**日期**: 2026-04-08  
**状态**: Accepted
