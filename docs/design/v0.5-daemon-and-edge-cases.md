# DoraMemory 设计方案 v0.5

**版本**: v0.5  
**日期**: 2026-04-08  
**状态**: 草稿（讨论中）  
**变更摘要**: /write 接口消失，改为文件监控守护进程；梳理边界问题

---

## 相对 v0.4 的核心变更

| 主题 | v0.4 | v0.5 |
|------|------|------|
| 写入方式 | POST /write，框架主动调用 | 文件监控守护进程，零侵入 |
| 对外接口 | /write + /recall + /remember | /recall + /remember（两个） |
| 接入成本 | 框架需要改代码 | 启动守护进程即可，零改造 |

---

## 系统本质

```
输入：文件系统事件（messages 落盘）
处理：后台 cron 压缩链
输出：MEMORY.md

对外接口只剩两个：
  POST /recall    agent 主动检索
  POST /remember  agent 主动标记
```

DoraMemory 启动后完全自治，agent 框架零配置、零改造。

---

## 守护进程架构

```
npx doramemory start

  ┌─────────────────────────────────┐
  │         DoraMemory Daemon       │
  │                                 │
  │  FSWatcher                      │
  │  监控配置的 watch paths          │
  │  文件变更 → 解析 → 写 second/   │
  │                                 │
  │  Cron                           │
  │  定时压缩各层 → 刷新 MEMORY.md  │
  └─────────────────────────────────┘
```

## 配置

```yaml
# ~/.doramemory/config.yaml

watch:
  - path: ~/.claude/projects/
    format: claude         # JSONL，每行一条消息
  - path: ~/.cursor/conversations/
    format: openai         # OpenAI message 格式
  - path: ~/my-agent/logs/
    format: openai

compression_model: claude-haiku-4-5  # 用便宜的模型做压缩
memory_file: ~/.claude/CLAUDE.md     # MEMORY.md 写入目标
```

format 做成插件式，新框架加一个 parser 即可。

## 边界问题与处理方案

### 1. 增量读取：文件被追加时不重复处理

messages 文件通常是增量追加的（JSONL append），守护进程需要记住已处理到的位置：

```
~/.doramemory/cursors/
  {文件路径的 hash}.cursor   # 存储已读取的 byte offset
```

文件变更时从 cursor 位置开始读，处理完更新 cursor。重启后从 cursor 恢复，不重复处理历史。

---

### 2. Session 边界检测

没有显式的 session 开始/结束信号，需要推断：

```
启发式规则：
  同一文件内，相邻两条消息时间差 > 阈值（默认 30 分钟）
  → 认为是新的 session

文件粒度：
  Claude Code 每个 project/conversation 是独立文件
  → 文件即 session，天然有边界
```

session_id = 文件路径 hash，稳定且唯一。

---

### 3. 并发写入：多个 session 同时活跃

```
Claude Code 多窗口、Cursor + Claude Code 同时用
→ 多个文件同时被监控
→ second/ 文件按时间戳命名，天然不冲突
→ 压缩时按 session_id 分组，互不干扰
```

唯一需要注意：MEMORY.md 的写入加文件锁，避免多个压缩任务同时写。

---

### 4. 冷启动：已有大量历史 messages

首次启动时，watch paths 下可能已有几个月的对话历史：

```
策略：不全量处理历史（代价太高）
      只处理最近 N 天（默认 7 天）的文件
      更早的历史视为"已压缩"，不补录 second/

配置项：
  cold_start_days: 7   # 冷启动时回溯天数
```

用户如果想补录更多历史，可以手动执行：
```
npx doramemory backfill --days 30
```

---

### 5. 文件格式解析失败

不同框架的 messages 格式可能有版本差异，解析失败时：

```
策略：跳过解析失败的条目，记录到 ~/.doramemory/errors.log
      不中断守护进程
      定期提示用户检查 errors.log
```

宁可丢几条消息，不能让整个系统崩溃。

---

### 6. Oversize 消息

单条消息超过阈值（默认 4k tokens），例如 tool result 返回大文件内容：

```
重要内容（LLM 判断）：
  → second/ 存原文（oversized: true）
  → 同时立即生成 minute/ 摘要

噪音内容（文件列表、日志等）：
  → 截断写 second/（truncated: true）
```

判断"重不重要"是唯一需要 LLM 的写入时操作，但可以做成异步、低优先级。

---

### 7. MEMORY.md 更新频率

更新太频繁浪费资源，太慢会导致 context 过时：

```
触发更新的条件（任一满足）：
  - 新的 second/ 文件写入后 5 分钟内无新消息（session 可能暂停）
  - cron 压缩任务完成
  - 用户执行 /remember

不触发更新的情况：
  - session 进行中（高频消息写入期间）
  - MEMORY.md 5 分钟内已更新过
```

---

### 8. 消息隐私

所有数据本地存储，但 messages 包含敏感信息：

```
默认行为：
  - 数据只在 ~/.doramemory/ 本地存储
  - 压缩调用的 LLM 可配置为本地模型（ollama）
  - 不做任何云同步

未来可选：
  - 加密存储 ~/.doramemory/
  - 支持 .doraignore 排除特定 path 的监控
```

---

### 9. Session Resume 的视图重建时机

长存续 session 恢复时需要实时重建 session-aware 视图（一次 LLM 调用）。

触发条件：
```
last_active_at 距今 > 1 小时（可配置）
→ 守护进程检测到同一 session 文件重新活跃
→ 异步重建 session_view
→ 写入 ~/.doramemory/sessions/{session_id}/view.md
→ agent 可通过 /recall 拿到，也可配置自动追加进 MEMORY.md
```

## 最终接口

```
POST /recall
  { query?, time_range?, max_tokens? }
  → MemoryChunk[]，支持语义检索和时间范围，sources 字段支持 drill-down

POST /remember
  { memory_id, flashbulb?, content? }
  → 标记 flashbulb 或修正 day/ 及以上层级的摘要内容
```

守护进程本身不暴露任何写入接口，写入完全由文件监控驱动。

---

## 完整数据流（终稿）

```
files 变更（messages 落盘）
  → FSWatcher 检测
  → 从 cursor 增量读取
  → 写入 second/（纯 IO）
  → oversize 消息异步处理

后台 cron
  → second→minute→hour→day→week→month→year
  → 压缩时 LLM 识别 flashbulb → core/recent/
  → recent/ 超限 → 元压缩 → core/identity.md
  → 每次完成后刷新 MEMORY.md

MEMORY.md（三层结构，体积有界）
  → agent 在 session 开始时自动读取
  → session resume 时追加 session_view

agent 主动操作
  → /recall：语义 or 时间范围检索
  → /remember：标记 or 修正
```

## Decision Log

### DDR-018: 去除 /write 接口，改为文件监控
**决策**: DoraMemory 守护进程监控 messages 文件变更，零侵入框架。  
**理由**: messages 本来就会落盘，监控文件变更比要求框架主动调用接口侵入性更低，接入成本为零。  
**日期**: 2026-04-08  
**状态**: Accepted

### DDR-019: 用 cursor 文件追踪增量读取进度
**决策**: 每个被监控文件维护一个 byte offset cursor，重启后恢复。  
**理由**: 避免重复处理历史消息，保证幂等性。  
**日期**: 2026-04-08  
**状态**: Proposed

### DDR-020: 冷启动只回溯最近 N 天
**决策**: 首次启动默认只处理最近 7 天历史，更早的视为已压缩。  
**理由**: 全量处理历史代价不可控；用户可以手动 backfill。  
**日期**: 2026-04-08  
**状态**: Proposed
